# 공문접수 관리 시스템 — 배포 가이드

> **GitHub → Replit Republish** 방식으로 배포합니다.  
> DB는 **공문접수 전용 Supabase Postgre젝트**를 사용하며, 로컬·Replit이 **동일한 `DATABASE_URL`** 로 같은 DB에 접속합니다.

| 항목 | 값 |
|------|-----|
| GitHub | [https://github.com/plzkanu/official](https://github.com/plzkanu/official) |
| 기본 브랜치 | `master` |
| 로컬 FE | http://localhost:5173 |
| 로컬 BE | http://localhost:8000 |
| DB | Supabase PostgreSQL (`od_*` 테이블) |

## 1. 사전 준비 (최초 1회)

### 1-1. Supabase 테이블 생성

1. [Supabase Dashboard](https://supabase.com/dashboard) → **공문접수 전용 프로젝트**
2. **SQL Editor** → `supabase/apply-all-migrations.sql` 붙여넣기 → **Run**
3. **Table Editor**에서 테이블 확인:
   - `od_departments`, `od_users`, `od_documents`, `od_reception_counters`, `od_digital_stamp`
4. **Storage** → `document-attachments` 버킷 생성 확인 (`apply-all-migrations.sql`에 포함)

### 1-2. DATABASE_URL 복사

Supabase → **Settings → Database → Connection string → URI**

| 항목 | 설정 |
|------|------|
| 연결 방식 | **Session pooler** (포트 `5432`) |
| 사용자 | `postgres.프로젝트ID` 형식 |
| 호스트 | `aws-0-...` 또는 `aws-1-...` (**Dashboard에 표시된 값 그대로**) |

> **Direct connection** (`db.xxx.supabase.co`)은 Windows/회사망에서 **IPv6 DNS 오류**로 실패할 수 있습니다.  
> **Session pooler** URI를 사용하세요.

**비밀번호 특수문자 URL 인코딩**

| 문자 | 인코딩 |
|------|--------|
| `&` | `%26` |
| `*` | `%2A` |
| `@` | `%40` |
| `#` | `%23` |

예시 형식:

```env
DATABASE_URL=postgresql://postgres.abcdefgh:비밀번호@aws-1-ap-northeast-2.pooler.supabase.com:5432/postgres
```

### 1-3. 환경 변수 파일

**로컬:** `backend/.env` (Git에 올리지 않음)

```env
SECRET_KEY=랜덤-문자열
DATABASE_URL=postgresql://postgres.프로젝트ID:...@aws-1-ap-northeast-2.pooler.supabase.com:5432/postgres
SUPABASE_SERVICE_ROLE_KEY=Supabase Dashboard → Settings → API → service_role
```

> **service_role** 키는 Supabase **Settings → API**에서 확인합니다. 첨부·날인본 파일을 Storage에 저장하는 데 사용합니다.

`backend/.env.example`을 복사해 시작할 수 있습니다.

**Replit:** Tools → **Secrets**에 동일 값 설정

| Secret | 필수 | 설명 |
|--------|------|------|
| `SECRET_KEY` | ✅ | JWT 서명 키 (Replit 기본 secret `SESSION_SECRET`도 인식) |
| `DATABASE_URL` | ✅ | Supabase Session pooler URI |
| `SUPABASE_SERVICE_ROLE_KEY` | ✅ | Supabase Storage (첨부·날인본) |
| `SUPABASE_URL` | | 생략 가능 (`DATABASE_URL`에서 자동 추론) |
| `SMTP_*` | | 메일 발송 (미설정 시 콘솔 출력) |
| `SUPABASE_SSL_VERIFY` | | TLS 오류 시 `0` (내부망 한정) |

### 1-4. GitHub · Replit (최초 1회)

```bash
git remote add origin https://github.com/plzkanu/official.git
git push -u origin master
```

Replit → **Import from GitHub** → `plzkanu/official`

## 2. 내 PC에서 테스트하는 방법

### 2-1. 백엔드 설정 (최초 1회)

```powershell
cd d:\dev\official_doc\backend
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
```

`backend\.env`를 열어 `SECRET_KEY`, `DATABASE_URL`을 입력합니다.

### 2-2. 서버 실행 (매번)

**터미널 1 — 백엔드**

```powershell
cd d:\dev\official_doc\backend
venv\Scripts\activate
uvicorn app.main:app --reload --port 8000
```

기동 성공 로그:

```
INFO:     Application startup complete.
```

**터미널 2 — 프론트엔드**

```powershell
cd d:\dev\official_doc\frontend
npm install
npm run dev
```

### 2-3. 접속 · 확인

| URL | 기대 결과 |
|-----|-----------|
| http://localhost:5173 | 로그인 화면 |
| http://localhost:8000/api/health | `"database":"connected"` |

health 응답 예:

```json
{"status":"ok","production":false,"database":"connected"}
```

`"database":"error"`이면 [문제 해결](#4-문제-해결) 참고.

### 2-4. 테스트 계정

서버 **최초 기동** 시 DB에 사용자가 없으면 자동 생성됩니다.

| 아이디 | 비밀번호 | 역할 |
|--------|----------|------|
| admin | admin1234 | 관리자 |
| registrar | reg1234 | 접수담당 |
| leader | leader1234 | 경영지원팀장 |
| hr_user | hr1234 | 담당부서 |

### 2-5. (선택) 프로덕션과 동일한 단일 서버

```powershell
cd d:\dev\official_doc
npm install
npm run build
$env:SECRET_KEY="local-test-secret"
$env:DATABASE_URL="postgresql://..."
$env:STATIC_DIR="../frontend/dist"
npm run start
```

→ http://localhost:8000

## 3. Replit으로 배포하는 방법

### 3-1. 아키텍처

```
로컬 수정 → git push → Replit Shell (동기화+빌드) → Republish
                              ↓
                    FastAPI + frontend/dist (단일 PORT)
                              ↓
                    Supabase PostgreSQL (동일 DATABASE_URL)
```

| 계층 | 기술 |
|------|------|
| 프론트 | React + Vite → `frontend/dist` |
| API | FastAPI `/api/*` |
| DB | Supabase (`od_*`) |
| 첨부·날인본 | **Supabase Storage** (`document-attachments` 버킷) |
| 접수도장 | **Supabase DB** (`od_digital_stamp`) |

### 3-2. Replit Secrets (필수)

| Secret | 값 |
|--------|-----|
| `SECRET_KEY` | JWT용 랜덤 문자열 (`SESSION_SECRET` 이름으로 등록해도 됨) |
| `DATABASE_URL` | 로컬 `backend/.env`와 **동일** |
| `SUPABASE_SERVICE_ROLE_KEY` | 로컬 `backend/.env`와 **동일** |

`.replit`에 `STATIC_DIR=../frontend/dist`가 설정되어 있습니다.

### 3-3. 최초 배포

Replit **Shell**:

```bash
git fetch origin master && git reset --hard origin/master && npm install && npm run build
```

→ **Deploy → Republish**

### 3-4. 코드 수정 후 재배포 (매번)

> ⚠️ **Republish만 하면 로컬 수정이 반영되지 않습니다.** 아래 ① push → ② Shell → ③ Republish 순서를 반드시 지키세요.  
> 첨부파일 Storage 코드가 GitHub에 없으면 Replit은 계속 `./uploads/` 구버전으로 동작합니다.

**① 로컬 push**

```bash
git add .
git commit -m "재수정"
git push origin master
```

**② Replit Shell**

```bash
git fetch origin master && git reset --hard origin/master && npm install && npm run build
```

**③ Republish** → 배포 URL에서 `/login` 확인

### 3-5. Replit 실행 설정 (`.replit`)

| 항목 | 값 |
|------|-----|
| Build | `npm run build` |
| Run | `npm run start` |
| Target | Cloud Run |
| Modules | nodejs-22, python3 |

빌드 로그 정상 예:

```
[install-python-deps] pip install backend/requirements.txt
[build] frontend vite build
```

### 3-6. 배포 후 확인

#### `/api/health` 확인 방법

**브라우저 주소창**에 배포 URL 뒤에 `/api/health` 를 붙여 접속합니다. (로그인 불필요)

| 환경 | URL 예시 |
|------|----------|
| **Replit 배포** | `https://official-doc.plzkanu.repl.co/api/health` ← Replit **Deployments** 탭에 표시된 URL + `/api/health` |
| **로컬 개발** | http://localhost:8000/api/health |
| **로컬 프로덕션 모드** | http://localhost:8000/api/health (`npm run start` 실행 시) |

Replit에서 배포 URL 확인: Replit 프로젝트 → **Deployments** (또는 Webview URL) → `https://....repl.co` 또는 `https://....replit.app`

**정상 응답 예 (최신 코드 + Secrets 설정 완료):**

```json
{
  "status": "ok",
  "production": true,
  "database": "connected",
  "database_error": null,
  "storage": "connected",
  "storage_error": null
}
```

| 필드 | 정상 | 문제 시 |
|------|------|---------|
| `database` | `"connected"` | `DATABASE_URL` Secrets 확인 |
| `storage` | `"connected"` | `SUPABASE_SERVICE_ROLE_KEY` + Storage 버킷 확인 |
| `storage` | `"skipped"` | **구버전 코드** — GitHub에 Storage 코드 미반영 |

> ⚠️ **`storage` 필드가 아예 없으면** Replit에 **구버전**이 배포된 상태입니다. 아래 [3-4 git push](#3-4-코드-수정-후-재배포-매번) 후 Republish 하세요.

| 확인 | 기대 결과 |
|------|-----------|
| `/login` | 로그인 화면 |
| `/api/health` | `"database":"connected"`, `"storage":"connected"` |
| 접수 등록 / 공문 수취 확인 | 역할별 메뉴 동작 |
| 첨부 · 날인 | 등록 후 접수대장에 **「첨부」** 버튼 표시, 수취 확인 가능 |

## 4. 문제 해결

### DB 연결

| 증상 | 원인 · 조치 |
|------|-------------|
| `getaddrinfo failed` (db.xxx.supabase.co) | Direct connection 사용 → **Session pooler** URI로 변경 |
| `tenant/user postgres.xxx not found` | pooler 호스트 오류 → Dashboard URI의 `aws-0`/`aws-1` **그대로** 사용 |
| `postgres.[project-ref] not found` | `[project-ref]` placeholder 미교체 → 실제 프로젝트 ID 입력 |
| `DATABASE_URL 환경 변수가 필요합니다` | `backend/.env` 또는 Replit Secrets 확인 |
| `relation "od_users" does not exist` | Supabase SQL Editor에서 `apply-all-migrations.sql` 실행 |
| `relation "od_digital_stamp" does not exist` | Supabase SQL Editor에서 `002_od_digital_stamp.sql` 또는 `apply-all-migrations.sql` 재실행 |
| Storage 업로드 실패 · `SUPABASE_SERVICE_ROLE_KEY` | Secrets/`backend/.env`에 service_role 키 설정, SQL Editor에서 `003_od_storage_bucket.sql` 실행 |
| 등록은 됐는데 접수대장에 **첨부파일 없음** | DB `file_path`가 `./uploads/...`이면 **구버전 배포** → 최신 코드 Republish. `documents/...`인데 없음이면 `/api/health`의 `storage` 확인 |
| 수취 시 **Storage에 첨부파일이 없어** | 위와 동일. Supabase Storage 버킷·service_role 키 확인 후 **손상된 건 삭제·재등록** |
| `/api/health` → `storage: error` | `SUPABASE_SERVICE_ROLE_KEY` 오류 또는 `document-attachments` 버킷 미생성 → SQL Editor에서 `003_od_storage_bucket.sql` 실행 |
| 예전 문서 날인본 없음 | Storage 도입 **이전**에 수취된 문서는 파일이 복구되지 않음 → 새로 접수·수취 |
| 비밀번호 오류 | 특수문자 **URL 인코딩** 확인 |

### 서버 · 포트

| 증상 | 조치 |
|------|------|
| `/api/health` 응답 없음 · 타임아웃 | 8000 포트에 uvicorn **중복 실행** → 아래 명령으로 정리 후 재시작 |
| `.env` 수정 후 반영 안 됨 | uvicorn **재시작** (Ctrl+C 후 다시 실행) |
| `Application startup failed` | 터미널 오류 로그 확인 → 대부분 DB 연결 문제 |

**8000 포트 정리 (Windows PowerShell)**

```powershell
Get-NetTCPConnection -LocalPort 8000 -ErrorAction SilentlyContinue |
  Select-Object -ExpandProperty OwningProcess -Unique |
  ForEach-Object { Stop-Process -Id $_ -Force -ErrorAction SilentlyContinue }
```

### Replit 배포

| 증상 | 조치 |
|------|------|
| 예전 코드가 배포됨 | Shell에서 `git reset --hard origin/master` 재실행 |
| 빌드 실패 | Shell에서 `npm run build`로 오류 확인 |
| `Can not perform a '--user' install` (pip) | `git pull` 후 재빌드. 계속되면 `rm -rf .venv && npm run build` |
| `403 Forbidden` (python-jose 등 pip) | 취약 버전 차단 → `requirements.txt` 최신 pull 후 `rm -rf .venv && npm run build` |
| 502 / crash loop / `Application startup failed` | Replit **Logs**에서 `RuntimeError:` 또는 `Application startup failed` 아래 한 줄 확인 |
| `SECRET_KEY 미설정` | Secrets에 `SECRET_KEY` 또는 Replit 기본 `SESSION_SECRET` 추가 |
| `SUPABASE_SERVICE_ROLE_KEY 미설정` | Secrets에 service_role 키 추가 (Storage 필수) |
| `frontend/dist 없음` | Shell에서 `npm run build` 후 Republish |
| `Application startup failed` + DB 오류 | `DATABASE_URL` 확인, Supabase SQL 마이그레이션 실행 |
| `프론트엔드 빌드 결과를 찾을 수 없습니다` | `npm run build` 후 Republish |
